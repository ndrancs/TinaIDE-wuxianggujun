# Git 设置说明

本文基于当前 Git 设置页源码整理，重点说明 **HTTPS 凭据、SSH 密钥、默认密钥、主机绑定** 这些和当前 UI 一一对应的能力。

## 先理解当前 Git 设置页

当前 Git 设置页不是老版本那种单页“Git 认证”，而是一个带 Tab 的页面：

- **HTTPS**
- **SSH**

也就是说，现在的 Git 设置已经把两类认证方式彻底拆开了。排障时先分清你到底在用哪一种。

## HTTPS 区域能做什么

当前 HTTPS 区域支持：

- 新增凭据
- 查看已保存主机列表
- 编辑已有凭据
- 删除已有凭据

列表里的每条记录，核心上对应的是一个 **host → username / token** 配置。

## 新增 HTTPS 凭据时要填什么

当前编辑器里主要有三个字段：

- Host
- Username
- Token

### Host 可以怎么填

源码里会对 Host 做归一化处理，所以你不一定非要只填裸域名。

这些输入都可能被解析成主机名：

- `github.com`
- `https://github.com`
- `github.com/user/repo`

最终真正保存时，会尽量提取成 host 本身。

如果你填进去的是空值或无法解析的内容，就会被判定为无效。

### Username 不填会怎样

当前实现里，如果 Username 留空，会自动回退成：

- `oauth2`

所以很多基于 Token 的 HTTPS 仓库，即使你不填用户名，也不至于完全不可用。

### Token 是必填吗

- **新增凭据**：必须填写 Token
- **编辑已有凭据**：可以留空，此时会沿用旧 Token

这一点很重要，因为很多人修改用户名时，不想重新抄一遍 Token。

## 已有 HTTPS 凭据怎么看

当前列表会展示：

- Host
- Username
- “已配置”状态

如果点进去编辑，就是在修改该主机对应的凭据，而不是新增第二条同 host 的记录。

## 什么时候适合用 HTTPS

适合场景：

- 快速接 GitHub / GitLab / Gitee
- 你已经有现成的 Personal Access Token
- 不想折腾 SSH 密钥

如果你只是偶尔拉取、推送一次，HTTPS 往往是最省心的方案。

## SSH 区域能做什么

当前 SSH 区域分成几块：

- 生成密钥
- 导入密钥
- 设置默认密钥
- 查看密钥列表
- 配置主机绑定

这已经不是只有“导入一个私钥”这么简单，而是一个完整的多密钥管理方案。

## 生成 SSH 密钥

当前设置页支持直接生成：

- **Ed25519** 密钥

可填写字段包括：

- 密钥名称
- Comment

生成成功后，密钥会出现在密钥列表里。

如果你只是想快速开始，这是最推荐的方式。

## 导入 SSH 私钥

当前设置页也支持从系统文件选择器导入私钥。

导入时会要求：

- 密钥名称
- Comment

源码里还会根据导入文件名自动给一个建议名称，所以通常不用从零开始手写。

## 默认密钥是什么

默认密钥表示：

- 当某个 Git 主机没有单独绑定时，优先使用哪把 SSH 密钥

你可以理解成“全局默认值”。

如果你只有一把 SSH 密钥，设置默认密钥通常就够了。

## 主机绑定是什么

这是当前实现里很重要、也很容易被忽略的能力。

主机绑定允许你为某个主机单独指定：

- Host
- Key Name
- Port

也就是说，即使全局默认密钥是 A，你也可以让：

- `github.com` 用 A
- `gitlab.company.local` 用 B
- 某个自建仓库走自定义端口和另一把密钥

这对多账号、多平台、多公司仓库并存时非常实用。

## 什么时候只设默认密钥就够了

如果你符合下面情况：

- 只有一把常用 SSH 密钥
- 所有仓库基本都走同一个账号
- 没有特殊端口和特殊主机

那你通常只需要：

1. 生成或导入密钥
2. 把它设成默认密钥

不一定要再配主机绑定。

## 什么时候必须用主机绑定

这些场景建议直接配主机绑定：

- GitHub 和公司 GitLab 用不同账号
- 某些主机走非 22 端口
- 同一台设备上同时管理多把 SSH 密钥

## 密钥列表里能做什么

当前密钥列表支持：

- 查看密钥名称
- 查看 comment 或类型信息
- 标识默认密钥
- 打开详情

在详情里还能进一步：

- 读取公钥
- 复制公钥
- 删除密钥

如果你要把公钥贴到 GitHub / GitLab / Gitee，最方便的入口就是这里的“复制公钥”。

## HTTPS 还是 SSH 怎么选

可以这样理解：

- **HTTPS**
  - 上手快
  - 配 Token 就能用
  - 适合轻量和临时场景
- **SSH**
  - 长期更稳
  - 更适合多仓库、多主机、多账号
  - 一次配好后，后续体验更接近桌面环境

## 常见问题

### 为什么我填了仓库地址，但 HTTPS 还是不生效

先检查：

1. Host 是否被正确提取成主机名
2. Token 是否真的有对应权限
3. 当前仓库 remote 是不是 HTTPS 地址

如果 remote 本身是 SSH 地址，那 HTTPS 凭据当然不会参与。

### 编辑 HTTPS 凭据时，Token 可以留空吗

可以，但前提是这条凭据本来就已经存在。

当前实现会在编辑时沿用旧 Token。

### 为什么我配了默认密钥，但某个主机还是不对

很可能是这个主机需要：

- 单独绑定另一把密钥
- 或者需要特殊端口

这时候请直接用 **主机绑定**，不要只改默认密钥。

### 导入了 SSH 私钥后，还要做什么

通常建议继续做两步：

1. 设默认密钥
2. 如有需要，再配主机绑定

否则某些仓库在实际连接时，仍然可能选不到你刚导入的那把钥匙。
