# 编译器设置说明

本文基于当前编译器设置页源码整理，重点说明优化等级、运行模式、格式化、工具链、sysroot 和重新部署开发环境这些入口。

## 编译器设置页现在包含什么

当前页面可以分成几块：

- 编译选项
- CMake 配置
- 代码格式化
- 工具链管理
- 工具链 / 开发环境

其中一部分入口只有在启用了 Linux 开发环境后才会显示。

## 编译选项

当前这部分主要有两项：

- **优化等级**
- **编译线程数**

### 优化等级

当前支持：

- `O0`
- `O1`
- `O2`
- `O3`

一般可以这样理解：

- `O0`：更适合调试
- `O2 / O3`：更偏性能

如果你遇到“调试时变量看不清”或“行为和源码不一致”，先看看是不是优化等级设得太高。

### 编译线程数

当前滑块范围是：

- 1 到 8

如果设备性能一般、发热明显或构建时卡顿，可以先把线程数适当调低。

## CMake 配置

这里包含：

- CMake 运行模式
- clang-format 运行模式
- Make 运行模式
- CMake Build Type
- CMake Generator
- Parallel Jobs

### 运行模式

当前运行模式选项本质上只有两种：

- `native`
- `proot`

但 `proot` 只有在当前启用了 Linux 开发环境时才会出现在选项里。

适合这样理解：

- **native**：更接近宿主环境
- **proot**：更适合依赖 Linux 工具链的场景

### Build Type

当前支持：

- Debug
- Release
- RelWithDebInfo
- MinSizeRel

如果你不知道选什么：

- 需要调试：优先 **Debug**
- 日常发布或性能测试：再考虑 **Release**

### Generator

当前支持：

- Unix Makefiles
- Ninja

如果项目没有特殊要求，按当前工程兼容性选择即可；不要为了“看起来高级”随便切。

### Parallel Jobs

当前是独立于“编译线程数”的另一个并行配置。

可以简单理解为：

- 编译线程数：更偏全局编译偏好
- CMake Parallel Jobs：更偏 CMake 构建并行度

如果你已经把构建打满机器，再继续把两个值都调高，未必会更快。

## 代码格式化

这里当前主要是：

- **Format Style**

它控制的是代码格式化时所采用的预定义风格，而不是单纯的“开/关”。

如果你觉得格式化结果不符合预期，优先先看这里，而不是先怀疑 clang-format 本身坏了。

## 工具链管理

这是当前页面最关键的一组入口之一，主要包括：

- 当前激活工具链
- 当前 NDK Runtime
- Sysroot 状态
- 导入工具链
- 导入 Sysroot

### 当前激活工具链

页面会显示当前已激活的工具链；如果没有可用工具链，则会显示未安装状态。

点击后可以从已安装工具链列表中切换。

### 当前 NDK Runtime

这里选择当前架构下使用的 Android sysroot / NDK runtime。

它会同时影响：

- 编译参数里的 `--sysroot`
- C++ 标准库头文件路径
- 运行时注入的 `libc++_shared.so`
- 导出 APK 时打包的 `libc++_shared.so`
- LSP 看到的 C/C++ 头文件环境

如果第三方库是用不同 NDK 版本构建的，可以导入对应 sysroot 后切换到它，避免 C++ runtime 不匹配。

### Sysroot 状态

这里是状态展示项，告诉你当前激活的 NDK Runtime / sysroot 是否已经安装。

如果构建链路总是缺系统头文件或平台相关资源，这里值得先看。

### 导入工具链

当前实现支持从文件导入工具链压缩包。

源码里会识别这些归档类型：

- `.tar.gz`
- `.tar.xz`
- `.tar`

导入过程会经历这些阶段：

- 解压
- 校验
- 安装
- 注册
- 切换为当前工具链

如果导入失败，问题往往不是“设置没保存”，而是压缩包本身结构不符合要求。

### 导入 Sysroot

这和导入工具链是分开的。导入成功后会注册为一个可切换的 NDK Runtime，并自动切换为当前选中项。

也就是说，你即使已经有工具链，也可能还缺对应架构的 sysroot。

如果导入工具链后依然存在构建环境不完整的问题，别忘了检查 sysroot。

## 重新部署开发环境

当 Linux 开发环境启用后，页面底部会出现：

- **重新部署开发环境**

当前实现里，这个动作会走完整的清理 + 初始化流程。

适合场景：

- PRoot 环境损坏
- 终端、调试、构建链路一起异常
- 你怀疑当前环境已经脏掉

不适合场景：

- 只是想微调一个编译参数
- 单个项目配置有误

## 哪些入口是“只看状态”，哪些是真正可改

可以这样区分：

- **状态类**
  - 当前工具链
  - Sysroot 状态
- **可修改类**
  - 优化等级
  - 编译线程数
  - 运行模式
  - Build Type
  - Generator
  - Parallel Jobs
  - Format Style
  - 导入工具链 / Sysroot
  - 重新部署环境

如果你只是想知道“为什么不能点”，先分清它是不是状态项。

## 常见问题

### 为什么我看不到 `proot` 运行模式

因为当前没有启用 Linux 开发环境时，`proot` 选项本来就不会出现在运行模式列表里。

### 工具链导入成功了，为什么还是不正常

先检查：

1. 当前激活工具链是否真的切过去了
2. Sysroot 是否已安装
3. 当前项目构建参数是不是仍然不匹配

### 什么时候该直接重新部署环境

适合直接重部署的情况：

- 终端、调试、构建都一起异常
- PRoot 环境明显损坏
- 你不确定问题是工具链、sysroot 还是 Linux 环境本身
